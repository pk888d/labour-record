# Implementation Guidelines

## 1. Product Rules

- Treat the database as the system of record.
- Treat DOCX files as import/export artifacts, not the primary data store.
- Keep hospital and shop workflows separate where the law/form structure differs.
- Use the existing templates and filled examples as acceptance references.
- Prefer explicit monthly cycles over free-floating records.

## 2. UI Guidelines

- Build the first screen as the active Kanban board, not a marketing dashboard.
- Use columns for workflow stages and cards for statutory forms.
- Show validation errors directly on cards and inside form screens.
- Keep data-entry grids dense and keyboard-friendly.
- Use filters for establishment, month, assigned user, and status.
- Use badges for form type, due state, validation count, and export state.
- Avoid decorative UI; this is an operational compliance tool.

## 3. Data Entry Guidelines

- Reuse employee master fields wherever possible.
- For monthly data, snapshot employees into the cycle so historical documents do not change when master data changes later.
- Support `Nil` as a deliberate value where forms require it, but store numeric zero as numeric zero.
- Store dates as dates, not text.
- Store money as decimal values, not floating point.
- Track manual overrides with reason and user.

## 4. Calculation Guidelines

Wage records should calculate:

- `gross_wages = basic + dearness_allowance + hra + other_allowances + overtime_wages + leave_wages`
- `total_deductions = pf + esi + labour_welfare_fund + advance_recovered + fine_or_damage_deduction + other_deductions`
- `net_wages = gross_wages - total_deductions`

Attendance should calculate:

- Days worked
- Leave days
- Absent days
- Wage-counted days

Leave should calculate:

- Opening balance
- Earned during period
- Availed during period
- Closing balance

## 5. Export Guidelines

- Keep template files versioned.
- Do not overwrite template files.
- Generated file naming pattern:
  - `{establishment_slug}_{YYYY_MM}_{form_code}_v{version}.docx`
- Store generated document metadata:
  - form task
  - cycle
  - generated by
  - generated at
  - template version
  - file path
- Export only approved cards unless admin override is used.

## 6. Audit Guidelines

Audit these events:

- Establishment created/updated/deactivated
- Employee created/updated/exited
- Monthly cycle created/locked
- Kanban status changed
- Reviewer comment added
- Card approved/rejected
- Approved data edited
- Document exported

Each audit entry should include:

- Entity type and ID
- Action
- Previous value where practical
- New value where practical
- User ID
- Timestamp

## 7. Engineering Guidelines

- Use migrations for all schema changes.
- Keep form definitions in code or database seed data, not scattered through UI components.
- Separate domain calculations from UI code.
- Write tests for calculations and workflow transitions before polishing UI.
- Keep generated documents out of source control.
- Keep original template samples in source control or protected storage.

## 9. Definition of Done

A feature is done when:

- Data is stored in structured tables.
- Required validations exist.
- UI handles empty, loading, error, and saved states.
- Audit logs are created for important changes.
- Role restrictions are enforced.
- Tests cover the core behavior.
- Generated documents are checked against the relevant template or filled sample.
